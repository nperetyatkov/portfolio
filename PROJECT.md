# PROJECT — Portfolio (Vibe Coding)

## 1. Цель проекта
Создать персональный сайт-портфолио на чистом HTML / CSS / JavaScript  
с архитектурой, пригодной для масштабирования, доработок и командной разработки.

Проект развивается итеративно, с сохранением рабочих версий через Git.

---

## 2. Текущий статус
- Legacy-версия сайта зафиксирована в GitHub
- Начат архитектурный рефакторинг
- разобран legacy файл style.css из ветки `main`
- Новая масштабируемая структура подключена для CSS и внедряется поэтапно (HTML и JS — в процессе)

---

## 3. Технологический стек
- HTML5 (семантическая разметка)
- CSS (структура: base / layout / components / pages)
- Vanilla JavaScript (модульный подход)
- Без фреймворков на старте
- Git + GitHub для контроля версий

---

## 4. Архитектурные принципы

### Общие правила
- Один файл = одна зона ответственности
- Компонентный подход без фреймворков
- Читаемость и поддерживаемость важнее краткости
- Код пишется так, как будто над проектом работает команда
- Любые изменения предлагаются поэтапно
- Перед написанием кода сначала обсуждается структура
- Новые файлы и папки добавляются только при явной необходимости

Запрещено:
- Переписывать legacy-код без отдельного этапа
- Предлагать фреймворки или сборщики без запроса
- Смешивать архитектурные слои

### Правило дробления файлов

Файлы создаются по зонам ответственности, а не по каждому отдельному элементу.

Один файл может содержать несколько сущностей, если:
- они работают вместе как единое целое;
- относятся к одному пользовательскому сценарию;
- имеют общую логику или визуальную ответственность.

Запрещено дробить код на отдельные файлы по принципу
«один элемент — один файл» без явной необходимости.

Новый файл создаётся только если:
- сущность переиспользуется в разных частях проекта;
- логика или стили могут развиваться независимо;
- текущий файл стал перегружен и потерял читаемость по смыслу.

Принцип:
**Не создавать новый файл, пока в существующем не стало тесно по смыслу.**

### CSS
- base — reset, переменные, типографика
- layout — крупные блоки (header, grid, footer)
- components — переиспользуемые элементы
- pages — стили конкретных страниц
- Все стили подключаются через `styles/main.css`

#### Порядок подключения
Стили подключаются сверху вниз:
1. base
2. layout
3. components
4. pages

Файлы из `pages` имеют право переопределять все выше.

### JavaScript
- Без глобальных переменных
- Логика разделена по модулям
- Инициализация всех модулей происходит в `scripts/main.js`
- Компоненты не знают друг о друге напрямую
- `features` — пользовательские сценарии и поведение,
  не относящиеся к одному конкретному компоненту

### Assets
- Изображения группируются по зоне ответственности (layout / components / pages)
- Имена файлов отражают назначение, а не внешний вид

## 5. Git и работа с версиями

### Ветки
- `main` — стабильная версия сайта (legacy, не изменяется)
- `new-structure` — активная разработка новой архитектуры

### Правила
- Один логический шаг = один коммит
- После каждого коммита сайт должен оставаться рабочим
- Рефакторинг и новые фичи не смешиваются в одном коммите
- Legacy-код не удаляется до завершения переноса

## 7. Правила изменений
- Существующий код не переписывается без явной причины
- Любые правки выполняются поэтапно
- Если есть риск поломки — сначала обсуждается стратегия
- Улучшения добавляются, а не внедряются внутрь legacy-логики

Если задача требует изменений в нескольких файлах:
- В начале работы определяется список файлов и цель правок в каждом
- Работа ведется поэтапно, по одному файлу за шаг
- В каждый момент времени предполагается, что открыт только один файл
- Переход к следующему файлу происходит только после завершения правок в текущем

Любой новый код должен сопровождаться указанием:
- в какой файл он добавляется
- в какое место (перед / после / внутри какого блока)
- что именно редактируется, если код заменяет существующий

## 10. Формат совместной работы (обязательный)

Работа над проектом ведётся строго по пошаговой схеме.
Один шаг = один файл.

### Общий порядок работы

1. Пользователь начинает новый диалог:
   - прикладывает `PROJECT.md` для контекста;
   - формулирует текущую задачу.

2. Ассистент:
   - анализирует задачу;
   - предлагает пошаговый план;
   - указывает, с каким файлом нужно работать первым.

3. Пользователь открывает указанный файл и предоставляет его содержимое.

4. Ассистент:
   - анализирует ТОЛЬКО этот файл;
   - объясняет, что в нём есть;
   - предлагает конкретные изменения.

5. Если для выполнения задачи требуется правка другого файла:
   - ассистент НЕ переходит к нему сам;
   - ассистент явно просит открыть следующий файл.

6. Переход к следующему файлу возможен
   ТОЛЬКО после завершения правок в текущем.

### Правила правок

- В каждый момент времени предполагается работа только с одним файлом.
- Ассистент не делает предположений о содержимом других файлов.
- Любая правка сопровождается пояснением:
  - что меняется;
  - зачем;
  - к какому архитектурному слою относится.
- При необходимости указываются ориентиры по строкам
  (если файл не менялся с момента открытия).

### Ограничения

Запрещено:
- предлагать правки сразу в нескольких файлах;
- перескакивать между файлами без явного запроса;
- вносить изменения «на будущее», не относящиеся к текущему шагу.

Принцип:
**Медленно, последовательно, контролируемо.**
